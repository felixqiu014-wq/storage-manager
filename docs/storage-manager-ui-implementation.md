# 存储管家 UI 实施说明

## 1. 文档目的

本文档用于沉淀“存储管家（Storage Manager / PVC 管理器）”当前前端原型的设计语言、交互逻辑、组件约束、状态定义与实施建议，便于后续：

- 继续做高保真 UI 落地
- 将原型拆分为正式业务组件
- 与设计稿、产品、研发进行统一对齐
- 作为后续联调与验收的参考标准

当前实现基于 React + Tailwind CSS + shadcn/ui + lucide-react。

---

## 2. 产品定位

存储管家是 Sealos 云桌面生态中的基础设施工具，目标是让用户更低认知成本地管理持久化存储卷（PVC）与卷内文件。

当前原型覆盖两种工作模式：

1. 全局管理模式
   用于跨命名空间查看 PVC 总览、容量分布、使用率、风险与资源浪费。

2. 场景联动模式
   由外部应用跳转进入某个 PVC，直接进入文件浏览器视图，对卷内文件执行浏览、编辑、上传等操作。

## 4. 页面结构总览

当前页面采用单页双视图结构，使用本地 React state 模拟 URL 驱动切换。

### 4.1 根容器

- 页面使用固定视口高度：`h-screen`
- 外层禁止整页滚动：`overflow-hidden`
- 主内容卡片在大屏下居中显示
- 小屏时优先让内部内容区滚动，而不是让底部分页等元素溢出到容器外

### 4.2 视图切换方式

通过 `selectedPvcId` 控制：

- `selectedPvcId === null`：展示全局存储大盘
- `selectedPvcId !== null`：展示文件浏览器

切换是组件级切换，不是路由跳转。

- 支持 `.dark` 模式变量切换
- radius、shadow、font 也已纳入变量体系
- Tailwind 配置已映射到这些 token

已接入的核心 token 包括：

- `background`
- `foreground`
- `card`
- `card-foreground`
- `primary`
- `primary-foreground`
- `secondary`
- `secondary-foreground`
- `muted`
- `muted-foreground`
- `accent`
- `accent-foreground`
- `destructive`
- `border`
- `input`
- `ring`
- `chart-1 ~ chart-5`

## 5.3 圆角体系

统一使用主题变量派生圆角：

- `radius-sm`
- `radius-md`
- `radius-lg`
- `radius-xl`

在页面中主要表现为：

- 主容器：大圆角
- 二级卡片：`rounded-2xl`
- 行级卡片：`rounded-xl`
- 按钮：`rounded-lg` 或 `rounded-xl`

## 5.4 阴影体系

当前已尽量改为使用语义阴影。

主要层次：

- 页面主容器：较柔和的大阴影
- 卡片与行卡片：轻量阴影，强调浮层关系而不喧宾夺主
- Dialog：中等强度阴影

## 5.5 字体与字号

当前使用系统无衬线字体栈。

主要文本层级：

- 页面主标题：约 28px，粗体
- 区块标题：较大号粗体
- 副标题/说明文字：14px 左右，`muted-foreground`
- 表格/列表正文：14px 左右
- 辅助信息：12px 左右

## 7. 状态色规范

## 7.1 PVC 状态

当前有四种业务状态：

1. 已挂载
2. 创建中
3. 异常
4. 未使用

## 7.2 状态展示形式

当前状态采用“状态灯 + 文本”的形式，不再使用高饱和 Badge。

状态灯规则：

- 已挂载：绿色圆点
- 创建中：黄色圆点
- 异常：红色圆点
- 未使用：灰色圆点

当前状态灯用显式颜色渲染，避免因主题 token 或类名生成失败而不显示。

## 7.3 使用率颜色

容量进度条根据使用率切换：

- 正常：蓝色
- 高位告警（>85%）：橙黄
- 危险（>95%）：红色

使用率数值颜色也同步变化。

## 20. 验收清单

后续开发或重构时，建议至少检查以下内容：

### 视觉

- 是否符合黑白中性主基调
- 主按钮是否均为黑底白字
- 次按钮是否均为白底黑字
- 是否仍有突兀的蓝色 focus 高亮

### 列表

- 三行记录是否在标准桌面视口内完整显示
- 小屏下分页是否仍在卡片内部
- 状态灯是否都显示
- 已挂载是否为绿色灯
- 创建中是否为黄色灯

### 交互

- 命名空间切换是否生效
- 浏览文件是否能切换到文件浏览器
- 返回大盘是否生效
- 双击目录是否能进入下一层
- 点击可编辑文件是否能弹出编辑器
- 扩容弹窗是否能正常打开

### 浮层

- Select 下拉宽度是否与触发器一致
- DropdownMenu 是否被遮挡
- Dialog 是否无蓝色 focus 外框

### 响应式

- 小屏下页脚不溢出
- 内部滚动行为正常
- 按钮与工具栏不换行错乱

## 21. 结论

当前这份原型已经具备以下特征：

- 具备明确的产品结构
- 具备可运行的双视图交互
- 已接入主题 token 体系
- 已形成较稳定的按钮、列表、状态与弹窗规范
- 已解决多轮视觉细节问题，包括状态灯、焦点态、小屏分页、下拉宽度、标题间距等

它已经足够作为正式 UI 实施的基础版本。

如果进入下一阶段，建议优先做：

1. 组件拆分
2. 设计 token 收敛
3. 文件浏览器与全局大盘的视觉统一
4. 真正接入后端数据与路由参数
