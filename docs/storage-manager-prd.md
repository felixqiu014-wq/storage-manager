# 存储管理（Storage Manager）产品需求文档 PRD

## 1. 文档信息
- 文档版本：v1.0
- 产出日期：2026-04-23
- 产品阶段：高保真前端原型（可用于研发拆解与测试用例设计）
- 目标读者：产品经理、前端/后端工程师、测试工程师、设计师

---

## 2. Problem Statement
当前用户在管理持久化存储卷（PVC）和卷内文件时，存在如下核心问题：

1. 无法在一个统一界面快速判断不同命名空间下存储卷的容量健康度与风险状态。  
2. 缺乏从“存储卷列表”到“卷内文件管理”的顺滑路径，日常排障和运维操作效率低。  
3. 文件删除、恢复、清空回收站等高风险操作需要更明确的确认机制与可追溯行为。  
4. 在小窗口场景下，内容密集页面容易出现布局溢出、可读性下降与操作误触。  
5. 缺少一份标准化、可执行的需求基线，导致后续开发和测试难以统一验收口径。

---

## 3. Solution
构建一个面向桌面端运维场景的“存储管理”工作台，覆盖“存储卷总览 - 文件管理 - 回收站”三大核心流程，并通过统一交互规则与确认机制降低高风险操作错误率。

解决方案包括：

1. 存储卷总览：支持命名空间筛选、关键指标展示、搜索、分页与状态识别。  
2. 文件管理：支持目录浏览、排序、文件上传、新建文件夹、在线编辑与删除（移入回收站）。  
3. 回收站：支持恢复单文件、清空回收站，所有危险操作有确认弹窗。  
4. 交互一致性：统一按钮、弹窗、状态色、错误提示与反馈时机。  
5. 响应式策略：小窗口下通过局部横向滚动、页面可滚动容器和紧凑布局保证可用性。

---

## 4. 产品目标与非目标

### 4.1 产品目标
1. 提供统一的 PVC 可视化管理入口。  
2. 缩短“发现风险 -> 定位存储卷 -> 处理卷内文件”的操作路径。  
3. 降低误删除和误清空风险。  
4. 为后续接入真实后端 API 打好前端交互和状态管理基线。  
5. 提供可直接落地的测试验收标准。

### 4.2 非目标（Out of Scope）
1. 多用户权限系统（RBAC）与细粒度权限控制。  
2. 真实对象存储/块存储后端联调。  
3. 文件内容 diff、版本管理、审计日志持久化。  
4. 批量多选文件操作（批量删除、批量恢复）。  
5. 移动端原生适配与触摸手势优化。

---

## 5. 目标用户与关键场景

### 5.1 目标用户
1. 平台运维工程师：关注存储容量、健康度、异常卷处理。  
2. 应用开发工程师：关注卷内配置文件修改、日志排查、快速上传。  
3. 测试/验证工程师：关注操作路径完整性、错误提示、边界行为一致性。

### 5.2 关键场景
1. 运维日常巡检：通过指标卡和状态快速识别高风险卷。  
2. 线上故障排查：进入卷内文件目录查看日志并编辑配置。  
3. 文件误删恢复：在回收站中恢复文件。  
4. 资源清理：确认后清空回收站。  
5. 小窗口办公：窗口尺寸变化时保持可读可操作。

## 6. 信息架构与导航

一级导航（侧栏）：
1. 存储卷  
2. 文件管理  
3. 回收站（带数量徽标）

主视图：
1. 存储卷视图（默认）  
2. 文件管理视图（需选择一个存储卷）  
3. 回收站视图

视图切换规则：
1. 从存储卷行点击“浏览文件”进入文件管理视图并绑定当前存储卷。  
2. 文件管理头部显示“当前存储卷：xxx”，可“返回存储卷列表”。  
3. 回收站可独立进入，不依赖当前存储卷。

## 7. 功能需求（Functional Requirements）

## 7.1 存储卷总览模块

### FR-001 命名空间筛选
1. 支持“全部命名空间 + 指定命名空间”切换。  
2. 切换后刷新指标卡数据、列表数据、分页状态。  
3. 切换命名空间后分页重置到第一页。

### FR-002 指标卡展示
1. 展示总分配容量。  
2. 展示已使用容量。  
3. 展示闲置资源（未使用卷容量总和）。  
4. 数据与当前筛选条件保持一致。

### FR-003 存储卷检索
1. 支持按存储卷名称和挂载实例名称模糊匹配。  
2. 空结果时展示明确空状态文案。  
3. 检索条件变化后分页重新计算。

### FR-004 存储卷列表与状态
1. 每条记录展示：名称、关联应用、命名空间、状态、容量与使用率、访问模式、操作。  
2. 状态包含：已挂载、创建中、异常、未使用。  
3. 使用率展示数值和进度条，按阈值展示不同风险色。  
4. 高风险卷支持扩容入口提示。

### FR-005 分页
1. 支持上一页/下一页。  
2. 展示当前页和总页数。  
3. 展示当前可见记录范围与总条数。  
4. 在不同窗口高度可调整每页条数策略（紧凑窗口减少条数）。

### FR-006 新建存储卷
1. 通过弹窗完成名称、命名空间、容量、访问模式录入。  
2. 校验规则：名称必填、容量为正数、命名空间下名称唯一。  
3. 成功后新增记录并关闭弹窗。

### FR-007 删除存储卷
1. 删除操作必须经过确认弹窗。  
2. 需输入完整存储卷名称作为二次确认。  
3. 已挂载卷禁用删除。  
4. 删除当前正在浏览的卷时，应回退到存储卷列表视图。

### FR-008 扩容存储卷
1. 提供扩容弹窗与容量滑块。  
2. 展示当前容量、目标容量与预计费用增量。  
3. 操作确认后进入后续扩容执行流程（当前原型可先做前端确认闭环）。

---

## 7.2 文件管理模块

### FR-101 当前存储卷标识
1. 文件列表区域显示“当前存储卷：xxx”。  
2. 同区域右侧提供“返回存储卷列表”按钮。

### FR-102 文件树浏览
1. 展示目录与文件层级结构。  
2. 支持目录展开/折叠。  
3. 支持双击目录进入。  
4. 高亮最近新增节点（短时高亮反馈）。

### FR-103 排序
1. 支持按名称、大小、最后修改时间排序。  
2. 名称支持“首字母/类型”两种模式。  
3. 点击相同列头可切换升序/降序。

### FR-104 新建文件夹
1. 通过弹窗输入名称与创建路径。  
2. 校验名称非空、同目录不重名。  
3. 成功后更新文件树并定位到目标路径。

### FR-105 上传文件
1. 通过弹窗选择本地文件并指定上传路径。  
2. 校验已选择文件、名称有效、同目录不重名。  
3. 成功后文件进入目标目录并可在列表中看到。  
4. 可编辑类型文件自动标记为可在线编辑。

### FR-106 在线编辑
1. 可编辑文件支持打开编辑弹窗。  
2. 弹窗展示文件名、内容编辑区与保存入口。  
3. 关闭弹窗不应影响其他页面状态。

### FR-107 删除文件（移入回收站）
1. 删除按钮必须弹出确认框。  
2. 确认后将文件从当前文件树移除。  
3. 同步新增回收站记录（名称、原路径、大小、删除时间、保留天数）。  
4. 若被删除文件正处于编辑弹窗中，需自动关闭编辑弹窗。

### FR-108 空状态
1. 当前目录无内容时展示空状态文案。  
2. 空状态下仍保留上传/新建入口。

---

## 7.3 回收站模块

### FR-201 回收站列表展示
1. 展示字段：名称、原位置、大小、删除时间、剩余保留时间、操作。  
2. 顶部展示“清空回收站”按钮。  
3. 不展示无效占位操作按钮。

### FR-202 恢复文件
1. 点击恢复需弹确认弹窗。  
2. 确认后从回收站移除该记录。  
3. 恢复后的归位逻辑后续可接真实文件系统（当前原型先保证回收站闭环）。

### FR-203 清空回收站
1. 清空操作需弹确认弹窗。  
2. 确认后清空全部回收站项。  
3. 清空后展示“回收站为空”状态。

---

## 7.4 响应式与布局模块

### FR-301 小窗口可用性
1. 根容器可滚动，避免内容被裁切。  
2. 列表区在窄窗口下允许局部横向滚动，不可出现元素重叠和断裂。  
3. 侧栏在中等宽度下支持紧凑宽度，优先保证主内容可用。  
4. 头部操作区在窄窗口允许换行。

### FR-302 视觉一致性
1. 主内容卡片采用内容自适应高度（Hug Contents），不贴底。  
2. 所有业务弹窗宽度统一收窄，保证视觉一致。  
3. 各页面标题、按钮、状态色保持统一规范。

---

## 8. 业务规则与异常处理

### 8.1 校验规则
1. 存储卷名称不能为空。  
2. 存储卷容量必须为正数。  
3. 同命名空间下存储卷名称不可重复。  
4. 文件夹同目录不可重名。  
5. 上传文件名在同目录内不可与现有文件/文件夹重名。  
6. 删除存储卷需输入精确名称。

### 8.2 风险操作保护
1. 删除存储卷、删除文件、恢复文件、清空回收站均需确认。  
2. 已挂载存储卷禁止删除。  
3. 所有确认弹窗均应提供“取消”与“确认”。

### 8.3 反馈策略
1. 表单校验失败需就地显示错误信息。  
2. 操作成功需即时反映在列表状态中。  
3. 空数据状态需有明确文案。

---

## 9. 非功能需求（NFR）

### 9.1 可用性
1. 常用操作路径不超过 2 次点击到达。  
2. 危险操作必须可撤销或可恢复（至少在回收站保留期内）。

### 9.2 一致性
1. 交互文案统一中文语义。  
2. 相同语义按钮使用一致视觉层级。

### 9.3 性能（原型阶段）
1. 视图切换无明显卡顿。  
2. 常规列表渲染无显著掉帧。  
3. 窗口缩放后布局重算稳定，无持续抖动。

### 9.4 可测试性
1. 每个模块具备明确输入/输出和可断言状态。  
2. 核心业务规则可通过自动化测试验证。

## 10. User Stories（详细）
1. 作为运维工程师，我希望能按命名空间筛选存储卷，这样我可以快速聚焦某个业务环境。  
2. 作为运维工程师，我希望看到总容量、已使用容量和闲置资源，这样我能快速判断资源效率。  
3. 作为运维工程师，我希望通过状态点区分“已挂载/异常/创建中/未使用”，这样我能优先处理风险卷。  
4. 作为运维工程师，我希望通过搜索直接定位卷名或实例名，这样我不需要逐条翻找。  
5. 作为运维工程师，我希望分页能显示总条数和区间，这样我能确认数据范围。  
6. 作为运维工程师，我希望高使用率卷有醒目提示，这样我能提前扩容避免事故。  
7. 作为运维工程师，我希望在卷行直接“浏览文件”，这样我可以快速进入排障路径。  
8. 作为运维工程师，我希望删除已挂载卷被禁用，这样可以避免误删在线业务数据。  
9. 作为运维工程师，我希望删除卷时输入卷名确认，这样可防止误操作。  
10. 作为运维工程师，我希望新建卷时校验重名，这样能避免配置冲突。  
11. 作为开发工程师，我希望在文件管理页明确看到当前存储卷名，这样不会在错误卷上操作。  
12. 作为开发工程师，我希望能一键返回存储卷列表，这样我可以快速切换目标卷。  
13. 作为开发工程师，我希望目录支持展开/折叠和双击进入，这样浏览效率更高。  
14. 作为开发工程师，我希望按名称、大小、时间排序，这样能快速找到目标文件。  
15. 作为开发工程师，我希望名称支持按“首字母/类型”排序，这样可按不同心智查找。  
16. 作为开发工程师，我希望可以在任意目录创建文件夹，这样可快速整理文件结构。  
17. 作为开发工程师，我希望创建文件夹时有重名校验，这样避免目录混乱。  
18. 作为开发工程师，我希望上传时可选择目标路径，这样不必上传后再移动。  
19. 作为开发工程师，我希望上传时检查重名冲突，这样避免覆盖风险。  
20. 作为开发工程师，我希望可编辑文件能在线修改，这样减少切换外部工具成本。  
21. 作为开发工程师，我希望删除文件前有确认弹窗，这样避免误删。  
22. 作为开发工程师，我希望删除文件后能进回收站而非立即消失，这样我能恢复误删内容。  
23. 作为开发工程师，我希望如果删除正在编辑的文件，编辑器自动关闭，这样状态不会错乱。  
24. 作为运维工程师，我希望回收站能展示原位置和删除时间，这样便于判断恢复价值。  
25. 作为运维工程师，我希望恢复文件前有确认弹窗，这样避免错误恢复。  
26. 作为运维工程师，我希望清空回收站有确认弹窗，这样避免不可逆误操作。  
27. 作为测试工程师，我希望每个错误输入都有明确错误文案，这样我能做稳定断言。  
28. 作为测试工程师，我希望空状态有固定文案，这样我能覆盖无数据场景。

30. 作为产品经理，我希望弹窗宽度统一且页面视觉一致，这样界面更专业。

## 11. Implementation Decisions
1. 使用单页多视图状态机管理“存储卷/文件管理/回收站”三大视图，保证切换成本低。  
2. 采用模块化状态结构：存储卷数据、文件树数据、回收站数据分离管理，便于后续替换为真实 API。  
3. 列表视图采用“摘要卡 + 列表卡 + 行卡片/表格”结构，提升信息密度与可读性。  
4. 高风险操作统一走确认弹窗模式，降低交互认知负担。  
5. 文件删除采用“移入回收站”而非硬删除，保留恢复路径。  
6. 排序、筛选、分页采用纯前端可预测逻辑，便于测试断言。  
7. 响应式优先保证可用性：当空间不足时优先局部滚动而非压缩到不可读。  
8. 表单校验就地显示错误，避免全局 Toast 淹没关键信息。  
9. 保留可扩展接口语义（如扩容确认、保存修改）以便后续接入后端执行流。  
10. 统一视觉 token、按钮层级与状态色规则，确保后续模块扩展一致性。

## 12. Testing Decisions

### 12.1 测试原则
1. 只验证外部行为和业务结果，不绑定实现细节。  
2. 所有高风险操作都必须覆盖“取消路径 + 确认路径”。  
3. 每条校验规则至少覆盖“合法输入 + 非法输入”两组用例。  
4. 响应式测试必须覆盖窄屏窗口，验证不重叠、不遮挡、可滚动。

### 12.2 建议测试层级
1. 单元测试：规则函数（排序、过滤、路径处理、重名校验、删除/恢复数据变换）。  
2. 组件集成测试：核心弹窗流程、文件删除到回收站流程、分页与筛选联动。  
3. 端到端测试：核心用户旅程（从存储卷筛选到文件删除再到回收站恢复）。  
4. 视觉回归测试：三大主视图在常见窗口尺寸下的布局稳定性。

### 12.3 核心验收用例清单（摘要）
1. 命名空间切换后指标与列表同步更新。  
2. 搜索关键字命中卷名与应用名。  
3. 新建存储卷成功和重名失败场景。  
4. 删除存储卷名称确认正确/错误场景。  
5. 文件夹创建重名校验。  
6. 上传文件未选择文件时报错。  
7. 文件删除确认后从文件列表消失并出现在回收站。  
8. 删除正在编辑文件时编辑器自动关闭。  
9. 回收站恢复确认后记录移除。  
10. 清空回收站确认后列表为空。  
11. 小窗口下存储卷列表与回收站列表可横向滚动。  
12. 页面在分屏场景下无关键按钮丢失或遮挡。

## 13. 验收标准（Definition of Done）
1. 三大模块（存储卷、文件管理、回收站）功能按本 PRD 完整可用。  
2. 所有高风险操作具备确认流程。  
3. 校验错误文案准确且可复现。  
4. 小窗口场景下页面无结构性崩坏。  
5. 核心流程测试用例通过率达到 100%。  
6. PRD 中“Out of Scope”内容未混入本期交付。

## 14. 风险与依赖
1. 当前为前端原型，未接真实后端时，部分动作仅能完成 UI 闭环。  
2. 接入真实文件系统后需明确恢复策略（恢复到原路径失败时的处理）。  
3. 如果后续引入权限系统，删除/恢复/扩容入口需按权限动态控制。  
4. 当数据量显著增加时，需评估虚拟列表与分页策略性能。
